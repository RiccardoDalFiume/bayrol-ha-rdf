# Release With GH

Use this skill to execute a safe, repeatable release flow for `RiccardoDalFiume/bayrol-ha-rdf`.

## Preconditions

- User confirms the target version tag (`vX.Y.Z`).
- User is authenticated (`gh auth status`).
- Release should normally be created from `main`.

## Workflow

Copy this checklist and track progress:

```text
Release progress:
- [ ] Confirm version tag with user
- [ ] Validate git and gh preflight state
- [ ] Update local main branch
- [ ] Create and push annotated tag
- [ ] Prepare descriptive release notes
- [ ] Create or edit GitHub release
- [ ] Verify release existence and summarize result
```

### 1) Confirm version

Ask for the exact version if missing. Use semantic versioning with `v` prefix.

Also confirm release type:

- final (default)
- draft
- prerelease

### 2) Validate preflight state

Run these checks:

```bash
git status
git branch --show-current
gh auth status
```

If `gh` is not found in PATH on Windows, discover and use explicit executable path:

```powershell
Get-ChildItem "C:\Program Files\GitHub CLI\gh.exe" -ErrorAction SilentlyContinue
```

Guardrails:

- Do not continue if auth fails.
- Do not create releases from dirty trees unless user explicitly confirms.
- Do not force-push tags.
- If current branch is not `main`, ask user before proceeding.

### 3) Sync branch

Run:

```bash
git checkout main
git pull origin main
```

### 4) Create and push tag

First verify whether the tag already exists:

```bash
git tag --list vX.Y.Z
```

Then create and push:

```bash
git tag -a vX.Y.Z -m "Release vX.Y.Z"
git push origin vX.Y.Z
```

### 5) Prepare descriptive release notes

Do not rely only on `--generate-notes` when the user asks for a quick user-facing summary.

Build a concise body with this template:

```markdown
## Highlights
- User-facing improvement 1.
- User-facing improvement 2.

## Technical changes
- CI/validation/dependency updates.
- Internal reliability fixes.

## Notes
Short compatibility or migration note if needed.

**Full Changelog**: https://github.com/RiccardoDalFiume/bayrol-ha-rdf/compare/<previous_tag>...vX.Y.Z
```

Use recent commits and changed files as input:

```bash
git log --oneline <previous_tag>..vX.Y.Z
```

### 6) Create or edit release

Default create command:

```bash
gh release create vX.Y.Z \
  --repo RiccardoDalFiume/bayrol-ha-rdf \
  --title "vX.Y.Z" \
  --generate-notes \
  --target main
```

Use `--notes` when a custom descriptive body is requested:

```bash
gh release create vX.Y.Z \
  --repo RiccardoDalFiume/bayrol-ha-rdf \
  --title "vX.Y.Z" \
  --notes "<custom_notes>" \
  --target main
```

Flags:

- append `--draft` when user wants draft.
- append `--prerelease` for beta/rc.

To update existing notes later:

```bash
gh release edit vX.Y.Z --repo RiccardoDalFiume/bayrol-ha-rdf --notes "<updated_notes>"
```

### 7) Verify and report

Run:

```bash
gh release view vX.Y.Z --repo RiccardoDalFiume/bayrol-ha-rdf
gh release list --repo RiccardoDalFiume/bayrol-ha-rdf --limit 5
```

Report back with:

- release tag
- release URL
- whether release is draft/prerelease/final
- whether notes are autogenerated or custom
- any errors and suggested follow-up actions

## Failure handling

- If `tag already exists`, stop and ask whether to use a new version.
- If `403` or permission errors occur, ask user to verify repository write access.
- If the release exists but is incorrect, ask whether to edit release notes or create a new patch tag.
